Давайте разберём случаи, в которых трейт не получится превратить в объект.
1. Методы, принимающие Self по значению

trait Consumer {
    fn consume(self); // ❌
} 

При динамической диспетчеризации dyn Trait хранится как указатель, а не как сам объект и его размер может быть неизвестен (?Sized). Передача self по значению потребовала бы знать точный размер, а его нет.
Методы должны принимать &self или &mut self, тогда компилятору не нужно знать размер типа.

2. Методы, возвращающие Self

trait CloneLike {
    fn duplicate(&self) -> Self; // ❌
} 

Компилятор не может реализовать impl CloneLike for dyn CloneLike, потому что Self здесь неизвестен — он не может «сконструировать новый dyn CloneLike», ведь не знает, какой конкретный тип надо клонировать (Dog? Cat?).
Метод duplicate требует знания точного типа при вызове, а dyn Trait стирает это знание.

3. Generic-методы

trait Converter {
    fn convert<T>(&self, val: T); // ❌
} 

Generic-методы нельзя поместить в vtable, потому что для них нужно сгенерировать отдельную реализацию под каждый T — это статическая диспетчеризация. А vtable — таблица фиксированных указателей, одинаковая для всех экземпляров типа.
Generic остаются только при статической диспетчеризации (impl Trait или <T: Trait>).


>>> Object-safe трейты <<<

Это трейты, в которых: 

    методы принимают self через ссылку (&self или &mut self);
    методы не возвращают Self;
    методы не generic.

Когда вы пишете &dyn Trait или Box<dyn Trait>, компилятор должен знать, как реализовать вызовы методов трейта через vtable. Для этого он внутренне создаёт что-то вроде:

impl Trait for dyn Trait {
    fn method(&self, ...) { 
        // вызвать через vtable
    }
} 

Чтобы метод попал в vtable, он должен быть вызываем без знания конкретного типа Self. Когда вы делаете dyn Trait, вы не знаете, это Dog или Cat.

Как обойти ограничения
Если метод не object safe, но вы хотите оставить его в трейте, можно сделать так:

trait Sensor {
    fn read(&self) -> i32;
    fn new() -> Self where Self: Sized; // ✅
} 

where Self: Sized как бы говорит: этот метод нельзя вызвать через dyn, он доступен только для конкретных типов.

Это позволяет сделать трейт частично object safe — его можно превратить в dyn Sensor, просто new() там будет недоступен.

dyn Trait — это указатель на данные + указатель на таблицу функций (vtable).
Чтобы таблица была возможна, все методы должны иметь одинаковую форму вызова, не зависеть от Self и не требовать мономорфизации под каждый тип.
Object safety — это гарантия того, что компилятор сможет реально создать impl Trait for dyn Trait.


Выберите все методы, которые делают трейт non-object-safe:

trait Example {
    fn clone_self(&self) -> Self;
    fn print(&self);
    fn new() -> Self;
    fn generic<T>(&self, x: T);
} 

Ответ:
fn clone_self(&self) -> Self
Верный ответ. Метод возвращает Self, а при dyn Trait конкретный тип неизвестен.

fn new() -> Self
Это верное утверждение. Возврат Self делает трейт non-object-safe.

fn generic<T>(&self, x: T)
Это верное утверждение. Методы с дженериками не могут быть object-safe, потому что T неизвестен при создании dyn Trait.
